`Account<T>{:rs}` requires `T: Pod{:rs}`, where Pod (plain old data) means the type is fixed-size, has `#[repr(C)]{:rs}` layout, and contains no padding holes. To make this constraint easy to satisfy, v2 ships a small set of alignment-1 wrappers that let most user data lay out cleanly without any manual padding.

## Integer wrappers

`PodU64{:rs}`, `PodU32{:rs}`, `PodU16{:rs}`, `PodU8{:rs}`, and the signed counterparts (`PodI128{:rs}`, `PodI64{:rs}`, `PodI32{:rs}`, `PodI16{:rs}`) store an integer as `[u8; N]{:rs}` with native little-endian encoding. Alignment 1 means no padding around the field, so wide integer fields sit flush against neighbors in a `#[repr(C)]{:rs}` struct:

```rust title="lib.rs" showLineNumbers=false
#[account]
pub struct Position {
    pub owner: Address,
    pub size: PodU64,
    pub entry_price: PodI128,
    pub leverage: PodU16,
}
```

Methods on each wrapper:

- `.get(){:rs}` reads the wrapped integer
- `From<native>{:rs}` constructs from a native integer
- `.checked_add{:rs}`, `.checked_sub{:rs}`, `.checked_mul{:rs}`, `.checked_div{:rs}` return `Option<Self>{:rs}`
- `.saturating_add{:rs}`, `.saturating_sub{:rs}`, etc. saturate at the type's bounds
- `.is_zero(){:rs}` short-circuit zero check
- `.neg(){:rs}` on signed types

## `PodBool{:rs}`

`PodBool{:rs}` is a single-byte boolean wrapper where any non-zero byte counts as `true` and a zero byte counts as `false`. Use it for flag fields in zero-copy structs to side-step the `bool{:rs}` validity-bit pitfall, where Rust treats it as undefined behavior to read a `bool{:rs}` whose underlying byte is anything other than `0` or `1{:rs}`:

```rust title="lib.rs" showLineNumbers=false
#[account]
pub struct Listing {
    pub seller: Address,
    pub active: PodBool,
    pub closed: PodBool,
    pub _pad: [u8; 6],
}
```

## `PodVec<T, MAX>{:rs}`

`PodVec<T, MAX>{:rs}` is a fixed-capacity vector with a `u16{:rs}` length prefix, stored inline in the account rather than behind a heap pointer. The on-disk layout is `[len:u16][pad:u16][T; MAX]{:rs}`, with `MAX{:rs}` baked into the type so the compiler knows the exact size at every site:

```rust title="lib.rs" showLineNumbers=false
#[account]
pub struct MultisigConfig {
    pub creator: Address,
    pub threshold: u8,
    pub bump: u8,
    pub label: PodVec<u8, 32>,
    pub signers: PodVec<Address, 10>,
}
```

The wrapper exposes the standard collection methods: `.len(){:rs}`, `.capacity(){:rs}`, `.as_slice(){:rs}`, `.as_mut_slice(){:rs}`, `.push(item){:rs}`, `.pop(){:rs}`, `.clear(){:rs}`, `.iter(){:rs}`, `.iter_mut(){:rs}`, and `.set_from_slice(&[T]){:rs}`. Because capacity is fixed at the type level, the wrapper never reallocates, and pushing past `MAX{:rs}` returns `Err(ErrorCode::CapacityError){:rs}` rather than growing the buffer.

`PodVec{:rs}` is the right choice when the upper bound on length is small and known ahead of time. When the data really is open-ended, reach for `BorshAccount<T>{:rs}` with regular `Vec<T>{:rs}` fields instead.

## `#[pod_wrapper]{:rs}` for enums

Plain Rust enums are not `Pod{:rs}` because the discriminant has bit patterns that the type system considers invalid. The `#[pod_wrapper]{:rs}` attribute solves this by deriving a transparent `u8{:rs}`-backed wrapper that validates the discriminant against the declared variants whenever the value is compared or converted:

```rust title="lib.rs" showLineNumbers=false
#[pod_wrapper]
#[repr(u8)]
pub enum OrderSide {
    Bid = 0,
    Ask = 1,
}

#[account]
pub struct Order {
    pub price: PodU64,
    pub side: PodOrderSide,
}
```

`PodOrderSide{:rs}` is the generated wrapper, where the operations `==`, `!=`, and `.into(){:rs}` each validate the underlying byte against the declared variants. Reading a stale or corrupted byte panics rather than returning an invalid value, which catches account-data tampering at the wrapper boundary instead of letting an invalid discriminant propagate into program logic.

For the rare cases where unvalidated access is the right choice, the raw `.0{:rs}` field is still available. This is useful for migrating an older variant set without panicking on values that pre-date the current declaration.
