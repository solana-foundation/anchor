Anchor CPI uses typed account structs and borrow-tracked account handles. `CpiContext<T>{:rs}` bundles typed accounts, the target program address, signer seeds, and remaining accounts. `CpiHandle<'a>{:rs}` carries account handles into the CPI.

## SPL CPI

`anchor-spl-v2` exposes account structs and helpers for common token instructions:

```rust showLineNumbers=false
use anchor_lang_v2::prelude::*;
use anchor_spl_v2::token::{self, accounts::TransferChecked};

let accounts = TransferChecked {
    from: ctx.accounts.from.cpi_handle_mut(),
    mint: ctx.accounts.mint.cpi_handle(),
    to: ctx.accounts.to.cpi_handle_mut(),
    authority: ctx.accounts.authority.cpi_handle(),
};

let cpi = CpiContext::new(ctx.accounts.token_program.address(), accounts);
token::transfer_checked(cpi, amount, decimals)?;
```

For PDA signatures, attach signer seeds:

```rust showLineNumbers=false
let seeds: &[&[&[u8]]] = &[&[b"vault", &[ctx.bumps.vault]]];
let cpi = CpiContext::new(token_program, accounts).with_signer(seeds);
```

## Generated program CPI

For Anchor programs, `#[program]{:rs}` emits CPI wrappers behind the `cpi` feature. A caller can use the callee's generated `program::cpi::accounts::*{:rs}` structs and wrapper functions:

```rust showLineNumbers=false
let cpi_accounts = callee::cpi::accounts::SetData {
    data: ctx.accounts.callee_data.cpi_handle_mut(),
    authority: ctx.accounts.authority.cpi_handle(),
};

let cpi_ctx = CpiContext::new(ctx.accounts.callee_program.address(), cpi_accounts);
callee::cpi::set_data(cpi_ctx, value)?;
```

The generated CPI account structs contain one `CpiHandle<'a>{:rs}` per account and implement `ToCpiAccounts<'a>{:rs}`. Wrapper functions pack instruction args and call `CpiContext::invoke{:rs}`.

<Callout variant="note" title="Current caveat">
  Generated CPI account wrappers currently skip accounts structs that contain `Option<_>{:rs}` or `Nested<_>{:rs}`. Those shapes need extra flattening and fallback logic. Runtime validation still supports them. This caveat is only about generated CPI wrappers.
</Callout>

## CpiHandle

`CpiHandle<'a>{:rs}` is intentionally narrow:

- `account.cpi_handle(){:rs}` creates a read-only handle
- `account.cpi_handle_mut(){:rs}` creates a writable handle and requires the account to be marked `mut{:rs}`
- `handle.address(){:rs}` returns the account address
- the handle does not deref to `AccountView{:rs}`

That missing `Deref{:rs}` is part of the safety story. It prevents accidental use with checked pinocchio CPI builders that would re-borrow the same account views.

## Why unchecked CPI is safe here

`CpiContext::invoke(data){:rs}` uses pinocchio's `invoke_signed_unchecked{:rs}`. The unchecked path skips a runtime borrow check, and Anchor replaces it with two constraints:

1. `CpiHandle<'a>{:rs}` borrows the typed account wrapper for as long as the CPI account struct exists, so Rust rejects concurrent typed mutable access.
2. pinocchio's account borrow state still catches stale raw-borrow cases after the CPI.

The goal is to turn common runtime borrow failures into compile-time errors while avoiding a check on every CPI.

## BorshAccount re-entry

If a CPI mutates an account that the current handler holds as `BorshAccount<T>{:rs}`, release and reacquire the borrow around the call:

```rust showLineNumbers=false
ctx.accounts.profile.release_borrow()?;
some_program::cpi::update_profile(cpi_ctx)?;
ctx.accounts.profile.reacquire_borrow_mut(ctx.program_id)?;
```

`Account<T>{:rs}` does not need this because its zero-copy fields read from the underlying buffer directly.
