---
title: IDL
description: How Anchor emits IDL metadata for instructions, accounts, types, events, constants, and account resolution.
---

The IDL is the contract between programs and clients. The emission path is trait-driven. Account wrappers and type wrappers expose metadata through `IdlAccountType{:rs}` and `IdlType{:rs}` under the `idl-build` feature.

## What the IDL includes

The IDL includes:

- instruction names, discriminators, args, and account lists
- account type definitions and discriminators
- plain type definitions from `#[derive(IdlType)]{:rs}`
- docs on instructions, account fields, and type fields
- PDA seed metadata when the macro can inspect seeds
- optional account flags
- relations from legacy `has_one{:rs}` syntax
- event definitions and serialization metadata
- constants from `#[constant]{:rs}`
- errors from `#[error_code]{:rs}`

`idl-build` pulls in IDL assembly machinery and should not be shipped in production SBF binaries.

## Account wrappers and IDL

Core wrappers such as `Signer{:rs}`, `Program<T>{:rs}`, `SystemAccount{:rs}`, `UncheckedAccount{:rs}`, and `Sysvar<T>{:rs}` surface account metadata through trait impls. SPL wrappers in `anchor-spl-v2` do the same from the downstream crate.

Custom wrappers should implement the IDL traits so the generated account list remains meaningful to clients.

## Seed metadata limits

IDL seed output is best for array-form seeds:

```rust showLineNumbers=false
#[account(seeds = [b"vault", authority.address().as_ref()], bump)]
pub vault: Account<Vault>,
```

Opaque expression seeds still work at runtime, but the IDL cannot always describe how a client should reproduce them:

```rust showLineNumbers=false
#[account(seeds = Vault::seeds(authority.address()), bump)]
pub vault: Account<Vault>,
```

Prefer array-form seeds for public instructions where generated clients should auto-derive PDAs.

## TypeScript status

The CLI can build IDL and TypeScript type files, but the TypeScript package story is still alpha. Current scaffolds are pinned to `@anchor-lang/core` `^1.0.0` until a stable v2 package is published. Treat generated TypeScript client surfaces as active development, not a stable npm contract.
