The TypeScript surface is currently an alpha integration path. The CLI builds IDL JSON and emits TypeScript type files. Current scaffolds use `@anchor-lang/core` `^1.0.0` while the stable v2 npm surface is still settling.

<Callout variant="warning" title="Alpha package surface">
  Current scaffolds are pinned to `@anchor-lang/core` `^1.0.0` until a stable v2 package is published. Treat TypeScript codegen as active development, and use Rust tests as the source of truth for exact instruction construction.
</Callout>

The practical rule during alpha is to use TypeScript for application integration and IDL inspection, while using generated Rust helpers and LiteSVM tests to prove exact instruction data, account ordering, PDA resolution, and optional-account behavior.

## Generated artifacts

`$ [34manchor[0m build{:ansi}` writes the client-facing artifacts under `target/`:

| Artifact | Path | Status |
| --- | --- | --- |
| IDL JSON | `target/idl/<program>.json` | The canonical cross-language description of the program. |
| TypeScript IDL type | `target/types/<program>.ts` | Useful for typed client experiments and app code. |
| Rust generated helpers | Program crate modules | The source of truth for instruction builders today. |
| JavaScript client package | `@anchor-lang/core` `^1.0.0` | Usable for IDL-backed experiments while the stable v2 package is settling. |

Regenerate the IDL artifacts with:

```ansi showLineNumbers=false
$ [34manchor[0m build
```

The generated IDL includes instruction names, discriminators, args, accounts, account types, events, constants, errors, optional account markers, and PDA seed metadata when the macro can inspect the seed list.

## Client program

A `Program{:ts}` is built from an IDL and a provider. Read-only flows can use any provider-shaped object with a `connection{:ts}`. Calls that send transactions need an `AnchorProvider{:ts}` or another provider that implements `sendAndConfirm{:ts}`.

In a frontend, wire the connection and wallet adapter into an Anchor provider before constructing the program:

```ts showLineNumbers=false
import { AnchorProvider, Program, setProvider } from '@anchor-lang/core'
import { useAnchorWallet, useConnection } from '@solana/wallet-adapter-react'
import type { Counter } from '../target/types/counter'
import idl from '../target/idl/counter.json'

const { connection } = useConnection()
const wallet = useAnchorWallet()

if (!wallet) {
  throw new Error('Wallet not connected')
}

const provider = new AnchorProvider(connection, wallet, {
  commitment: 'confirmed',
})
setProvider(provider)

export const program = new Program(idl as Counter, provider)
```

A read-only script can construct a program from a connection without a wallet:

```ts showLineNumbers=false
import { clusterApiUrl, Connection } from '@solana/web3.js'
import { Program } from '@anchor-lang/core'
import type { Counter } from '../target/types/counter'
import idl from '../target/idl/counter.json'

const connection = new Connection(clusterApiUrl('devnet'), 'confirmed')

export const program = new Program(idl as Counter, {
  connection,
})
```

This is useful for account queries, IDL-backed name lookup, and early integration tests that do not sign transactions.

## Invoke instructions

The methods builder remains the design center for TypeScript clients:

```ts showLineNumbers=false
const signature = await program.methods
  .initialize()
  .accounts({
    payer: wallet.publicKey,
    counter,
  })
  .rpc()
```

The builder chain has these pieces:

| Step | Purpose |
| --- | --- |
| `program.methods{:ts}` | Starts an instruction call from the IDL namespace. |
| `.initialize(...args){:ts}` | Selects the instruction and passes serialized args. |
| `.accounts({...}){:ts}` | Supplies accounts the client must provide. |
| `.signers([...]){:ts}` | Adds keypairs beyond the provider wallet when using `.rpc(){:ts}`. |
| `.rpc(){:ts}` | Builds, signs, sends, and confirms a transaction. |
| `.transaction(){:ts}` | Builds a transaction without sending it. |
| `.instruction(){:ts}` | Builds one instruction for manual transaction composition. |

Use `.instruction(){:ts}` when an app needs to compose an Anchor instruction with wallet-adapter flows, compute-budget instructions, or non-Anchor instructions:

```ts showLineNumbers=false
const ix = await program.methods
  .increment()
  .accounts({
    counter,
    authority: wallet.publicKey,
  })
  .instruction()
```

During alpha, compare important TypeScript-built transactions against the generated Rust instruction helper for the same instruction. The Rust helper currently has the final say on compact instruction discriminators, resolved account ordering, optional account sentinels, and generated PDA metadata.

## PDA seed metadata

Generated clients can only auto-derive PDAs when the IDL contains enough seed metadata. Array-form seeds are the best shape:

```rust showLineNumbers=false
#[account(seeds = [b"vault", authority.address().as_ref()], bump)]
pub vault: Account<Vault>,
```

Opaque expression seeds still validate at runtime, but clients may not know how to reproduce them:

```rust showLineNumbers=false
#[account(seeds = Vault::seeds(authority.address()), bump)]
pub vault: Account<Vault>,
```

For public client-facing instructions, prefer array-form seeds whenever generated TypeScript clients should derive addresses automatically.

## Account fetching

Account clients are exposed under `program.account{:ts}`:

```ts showLineNumbers=false
const counterAccount = await program.account.counter.fetch(counter)
const allCounters = await program.account.counter.all()
const manyCounters = await program.account.counter.fetchMultiple([
  counterA,
  counterB,
])
```

Use the account wrapper to choose the right decoding and filter offsets. Not every account starts with an Anchor discriminator followed by a borsh payload.

| Account wrapper | Client decoding note |
| --- | --- |
| `Account<T>{:rs}` | Zero-copy and Pod-backed by default. Fixed offsets come from the `#[repr(C)]{:rs}` layout after the Anchor discriminator. |
| `BorshAccount<T>{:rs}` | Variable-length borsh account data. |
| `Account<Mint>{:rs}` | SPL mint layout owned by the Token Program, with no Anchor discriminator. |
| `Account<TokenAccount>{:rs}` | SPL token-account layout owned by the Token Program, with no Anchor discriminator. |
| `InterfaceAccount<T>{:rs}` | Accepts Token or Token-2022 ownership. Token-2022 accounts may have TLV extension bytes after the base layout. |

For memcmp filters, calculate offsets from the actual wrapper layout. SPL token accounts start at byte offset `0`, while Anchor-discriminated program accounts start after the discriminator.

## Recommended workflow

Most projects should keep Rust integration tests as the exact execution harness:

1. Define the instruction and accounts in Rust.
2. Run `$ [34manchor[0m build{:ansi}` and inspect `target/idl/` plus `target/types/`.
3. Write or update a LiteSVM test using the generated Rust helper.
4. Build the TypeScript integration against the same IDL.
5. Keep TypeScript assumptions small until the alpha package lands.

This gives app developers a usable IDL and type file without making the TypeScript package carry guarantees that the alpha code has not made yet.

## Current limitations

Some features are still settling in downstream clients:

- compact one-byte instruction discriminators
- richer account-resolution metadata
- optional account sentinel handling
- generated CPI metadata
- opaque seed expressions
- Pod-backed account decoding
- SPL and Token-2022 interface-account surfaces

Use the IDL for integration planning and early experiments, but keep Rust tests and generated Rust account builders as the reference implementation until the TypeScript package catches up.
