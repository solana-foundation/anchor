title: Account types
description: The full lineup of account wrappers, including `Account<T>{:rs}`, `BorshAccount<T>{:rs}`, `Slab<H, Item>{:rs}`, and `Nested<T>{:rs}`.
---

v2 ships a small set of account wrappers, none of which carry an `<'info>{:rs}` lifetime parameter. pinocchio's account model is statically scoped at the program-handler boundary, which means wrappers can be passed around freely without the lifetime gymnastics that v1 required.

| Type | Use for |
| --- | --- |
| `Account<T>{:rs}` | Default for your program's data. Zero-copy. Requires `T: Pod{:rs}`. |
| `BorshAccount<T>{:rs}` | Data with `Vec{:rs}` / `String{:rs}` / enums. Deserializes on load, serializes on exit. |
| `Slab<H, Item>{:rs}` | Header + dynamic item tail. Zero-copy ledger / event-log accounts. |
| `Option<Account<T>>{:rs}` | Optional account slot. Sentinel when absent. |
| `Nested<T>{:rs}` | Compose `#[derive(Accounts)]{:rs}` structs. Inline expansion. |
| `Signer{:rs}` | Transaction signer. Validates `is_signer{:rs}`. |
| `Program<T: Id>{:rs}` | CPI targets (`Program<System>{:rs}`, `Program<Token>{:rs}`). Validates executable + program ID via `T: Id{:rs}`. |
| `SystemAccount{:rs}` | System-owned account. Owner check only. |
| `UncheckedAccount{:rs}` | Escape hatch. No validation. |
| `Sysvar<T>{:rs}` | `Sysvar<Clock>{:rs}`, `Sysvar<Rent>{:rs}`. Prefer the syscall (`Clock::get(){:rs}`) where possible. |

## `Account<T>{:rs}`

`Account<T>{:rs}` is the default wrapper for a program's own data and is zero-copy by default. The stored bytes match the struct's in-memory layout, so loading is a pointer cast and exit is a no-op rather than a serialize-then-write round-trip:

```rust title="lib.rs" showLineNumbers=false
#[account]
pub struct Counter {
    pub value: u64,
    pub bump: u8,
    pub _pad: [u8; 7],
}

#[derive(Accounts)]
pub struct Init {
    #[account(mut)]
    pub payer: Signer,
    #[account(init, payer = payer, seeds = [b"counter"], bump)]
    pub counter: Account<Counter>,
    pub system_program: Program<System>,
}
```

`#[account]{:rs}` enforces:

- `T: Pod{:rs}` (compile error if any field is not `Pod{:rs}`)
- `#[repr(C)]{:rs}` layout
- Auto-derived 8-byte discriminator (`sha256("account:" + name)[..8]{:rs}`)
- `Owner{:rs}` impl returning the program ID

Internally, `Account<T>{:rs}` is a type alias for `Slab<T, HeaderOnly>{:rs}`. The header-only specialization uses a zero-sized item tail that never materializes, so the two share a single implementation without paying for a list of items the wrapper doesn't need.

## `BorshAccount<T>{:rs}`

Use `BorshAccount<T>{:rs}` when an account's fields are variable-length, such as `Vec{:rs}`, `String{:rs}`, or enums with payload variants. The wrapper holds a borrow guard for the duration of the instruction, and `exit(){:rs}` serializes through that guard once the handler returns successfully. The `#[account(borsh)]{:rs}` form auto-derives `BorshSerialize{:rs}` and `BorshDeserialize{:rs}`, so the struct itself does not need a separate derive:

```rust title="lib.rs" showLineNumbers=false
#[account(borsh)]
pub struct Profile {
    pub owner: Address,
    pub name: String,
    pub friends: Vec<Address>,
}

#[derive(Accounts)]
pub struct UpdateProfile {
    #[account(mut)]
    pub owner: Signer,
    #[account(mut, address = profile.owner)]
    pub profile: BorshAccount<Profile>,
}
```

If a CPI mutates the same account through another path while a `BorshAccount<T>{:rs}` is held, call `release_borrow()?{:rs}` before the CPI and `reacquire_borrow_mut(program_id)?{:rs}` afterwards. `release_borrow{:rs}` first commits any in-memory changes back to the buffer so the CPI sees them, then drops the borrow guard. `reacquire_borrow_mut{:rs}` re-runs the owner, discriminator, and size checks against the post-CPI state and re-deserializes the data, so any changes the foreign program made are picked up cleanly.

## `Slab<H, Item>{:rs}`

`Slab<H, Item>{:rs}` is the underlying primitive for a header struct followed by a variable-length tail of items. The on-disk layout is `[disc:8][H][len:u32][pad to align_of<Item>][Item; capacity]{:rs}`, and capacity is computed from `view.data_len(){:rs}` at load time, so the same code adapts to whatever capacity the account was allocated with:

```rust title="lib.rs" showLineNumbers=false
#[derive(Pod, Zeroable, Copy, Clone)]
#[repr(C)]
pub struct OrderBookHeader {
    pub mint: Address,
    pub authority: Address,
}

#[derive(Pod, Zeroable, Copy, Clone)]
#[repr(C)]
pub struct Order {
    pub price: PodU64,
    pub size: PodU64,
}

#[derive(Accounts)]
pub struct PostOrder {
    #[account(mut)]
    pub authority: Signer,
    #[account(mut, has_one = authority)]
    pub book: Slab<OrderBookHeader, Order>,
}
```

The wrapper exposes the usual `Vec`-style methods (`try_push{:rs}`, `pop{:rs}`, `iter{:rs}`, `swap_remove{:rs}`, indexing, and friends), plus `resize_to_capacity{:rs}` under the `account-resize` feature. `try_push{:rs}` returns `Err(ProgramError::AccountDataTooSmall){:rs}` when the tail is full, since the wrapper never touches lamports on its own. Grow the account through `resize_to_capacity{:rs}` and pair it with a `top_up{:rs}` call to settle rent before pushing past the current capacity.

## `Option<Account<T>>{:rs}`

Wrapping any account type in `Option{:rs}` marks the slot as optional. The client passes the program ID as a sentinel value when the slot is absent, and the corresponding `bumps.field{:rs}` shifts from `u8{:rs}` to `Option<u8>{:rs}` so the handler can tell the two cases apart:

```rust title="lib.rs" showLineNumbers=false
#[derive(Accounts)]
pub struct Maybe {
    #[account(mut)]
    pub user: Signer,
    #[account(seeds = [b"profile", user.address().as_ref()], bump)]
    pub profile: Option<Account<Profile>>,
}
```

## `Nested<T>{:rs}`

`Nested<T>{:rs}` composes one `#[derive(Accounts)]{:rs}` struct inside another, which lets several instructions share a common validation block without copying its fields. The inner struct's accounts are reachable through `ctx.accounts.field.field_in_nested{:rs}`:

```rust title="lib.rs" showLineNumbers=false
#[derive(Accounts)]
pub struct AdminConfig {
    pub admin: Signer,
    #[account(has_one = admin, seeds = [b"config"], bump = config.bump)]
    pub config: Account<Config>,
}

#[derive(Accounts)]
pub struct Increment {
    pub admin: Nested<AdminConfig>,
    #[account(mut, seeds = [b"counter"], bump = counter.bump)]
    pub counter: Account<Counter>,
}

#[derive(Accounts)]
pub struct Reset {
    pub admin: Nested<AdminConfig>,
    #[account(mut, seeds = [b"counter"], bump = counter.bump)]
    pub counter: Account<Counter>,
}
```

`Nested<T>{:rs}` expands inline at the call site, so it does not consume an extra account slot in the instruction. The [`bench/programs/nested/anchor-v2`](https://github.com/solana-foundation/anchor/tree/anchor-next/bench/programs/nested/anchor-v2) program shows the pattern end-to-end.

## `Signer{:rs}`

`Signer{:rs}` validates that the account signed the transaction by checking its `is_signer{:rs}` flag, and it does not deserialize any account data. The signer's pubkey is available through `account.address(){:rs}`, and combining the wrapper with `mut{:rs}` adds an `is_writable{:rs}` check on top of the signature requirement.

## `Program<T: Id>{:rs}`

`Program<T: Id>{:rs}` is the wrapper used for CPI target accounts, parameterized by a marker type that implements `Id{:rs}`. The `programs{:rs}` module defines markers for the common cases (`System{:rs}`, `Token{:rs}`, `Token2022{:rs}`, `AssociatedToken{:rs}`, and `Memo{:rs}`); the prelude re-exports `System{:rs}`, `Token{:rs}`, and `Token2022{:rs}`, while `AssociatedToken{:rs}` and `Memo{:rs}` are reachable through `anchor_lang_v2::programs{:rs}`. At load time the wrapper validates that the account's address matches `T::id(){:rs}`, and (when the `guardrails` feature is on, which is the default) that `executable == true{:rs}`.

## `SystemAccount{:rs}`

`SystemAccount{:rs}` validates that the account is owned by the system program and nothing more. It carries no data and does not require a signature, which makes it the right choice for signaling that an account exists on chain but is not a typed Anchor account that the program manages.

## `UncheckedAccount{:rs}`

`UncheckedAccount{:rs}` is the escape hatch for situations where v2's built-in validation is too restrictive or doesn't apply. The wrapper performs no checks at all, so pair it with explicit derive-level constraints like `address = X{:rs}` or `owner = Y{:rs}`, or hand-rolled validation inside the handler, to keep the boundary safe.

## `Sysvar<T>{:rs}`

`Sysvar<T>{:rs}` provides account-form access to the runtime's sysvars, including `Clock{:rs}` and `Rent{:rs}`. Internally the wrapper validates that the passed address matches `T::SYSVAR_ID{:rs}` and then calls pinocchio's `Sysvar::get(){:rs}` syscall to populate the data, so the account's payload is never deserialized. Where the handler does not need the typed account view, prefer the syscall directly (`Clock::get(){:rs}` or `Rent::get(){:rs}`) so the caller can drop the sysvar account from the transaction's account list.

## Trait surface

Every account wrapper implements [`AnchorAccount{:rs}`](/docs/v2/extensibility/#anchoraccount). Third-party crates can ship new account types by implementing the trait themselves. See [`anchor-dynamic-account`](https://github.com/chen-robert/anchor-dynamic-account) for a worked example.
