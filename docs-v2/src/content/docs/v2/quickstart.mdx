This page walks through scaffolding a v2 counter program from initialization through tests. The overall flow matches v1 closely, with the v2-specific differences spelled out as they come up.

## Init a workspace

```ansi
$ [34manchor[0m init [2m--no-install[0m counter
$ [34mcd[0m counter
```

The `[2m--no-install[0m{:ansi}` flag skips the npm install for the JS test scaffold, which is fine because v2 ships a Rust-side `litesvm` test harness through `anchor-v2-testing`. You do not need the Node or Yarn toolchain to run a v2 program's tests.

## Hello-world program

Replace the contents of `programs/counter/src/lib.rs` with the following:

```rust title="programs/counter/src/lib.rs" showLineNumbers=false
use anchor_lang_v2::prelude::*;

declare_id!("CounterPLACEHOLDER1111111111111111111111111111");

#[program]
pub mod counter {
    use super::*;

    pub fn init(ctx: &mut Context<Init>) -> Result<()> {
        let counter = &mut ctx.accounts.counter;
        counter.value = 0;
        counter.bump = ctx.bumps.counter;
        Ok(())
    }

    pub fn increment(ctx: &mut Context<Increment>) -> Result<()> {
        ctx.accounts.counter.value += 1;
        Ok(())
    }
}

#[derive(Accounts)]
pub struct Init {
    #[account(mut)]
    pub payer: Signer,
    #[account(init, payer = payer, seeds = [b"counter"], bump)]
    pub counter: Account<Counter>,
    pub system_program: Program<System>,
}

#[derive(Accounts)]
pub struct Increment {
    #[account(mut, seeds = [b"counter"], bump = counter.bump)]
    pub counter: Account<Counter>,
}

#[account]
pub struct Counter {
    pub value: u64,
    pub bump: u8,
    pub _pad: [u8; 7],
}
```

A handful of details look different from v1:

- The prelude import becomes `use anchor_lang_v2::prelude::*{:rs}` rather than `use anchor_lang::prelude::*{:rs}`.
- Each handler takes `&mut Context<T>{:rs}` instead of `Context<T>{:rs}` by value, which lets constraint hooks and exit logic write back to the same context.
- The wrappers `Signer{:rs}`, `Account<Counter>{:rs}`, and `Program<System>{:rs}` no longer carry an `<'info>{:rs}` lifetime parameter, because pinocchio's account model is statically scoped at the program-handler boundary.
- The `#[account]{:rs}` attribute requires the inner struct to be `Pod{:rs}`, meaning fixed-size with `#[repr(C)]{:rs}` enforced by the macro. The `value: u64{:rs}` field plus `bump: u8{:rs}` and `_pad: [u8; 7]{:rs}` together keep the layout 8-byte aligned.

For variable-length data like `Vec<u8>{:rs}` or `String{:rs}`, switch the wrapper to [`BorshAccount<T>{:rs}`](/docs/v2/account-types/#borshaccount-t) instead.

## Build and test

```ansi
$ [34manchor[0m build
$ [34manchor[0m test
```

`$ [34manchor[0m test{:ansi}` runs the Rust integration tests under `programs/counter/tests/`. With the `profile` feature on `anchor-v2-testing`, register traces land under `target/anchor-v2-profile/<test>/` for downstream flamegraph or [debugger](/docs/v2/testing-and-debugging/#anchor-debugger) consumption.

## Step through with `[34manchor[0m debugger{:ansi}`

After running tests at least once so the trace files exist, launch the debugger TUI:

```ansi
$ [34manchor[0m debugger
```

From there, step through SBF instructions one at a time, scrub registers, and jump back to source. The full set of keybindings and modes is documented in [Testing and debugging](/docs/v2/testing-and-debugging/#anchor-debugger).
