---
title: Performance and optimizations
description: Where Anchor's binary and CU savings come from.
---

Anchor's performance gains are cumulative. No single change explains the full result. The savings come from many smaller choices that compose.

The main sources are:

- the pinocchio and `#![no_std]{:rs}` runtime path
- a smaller core derive that delegates account behavior to traits the compiler can optimize
- `Account<T>{:rs}` casting account bytes directly into Pod-backed fixed-size state
- macro-time PDA resolution when seeds are byte literals
- runtime PDA validation that can skip repeated on-curve checks for non-empty program-owned PDA accounts
- typed `CpiHandle<'a>{:rs}` values that let Anchor use pinocchio's unchecked CPI path without ordinary typed mutable aliasing
- wincode instruction and event encoding with a borsh-compatible wire configuration
- `#[event(bytemuck)]{:rs}` for fixed-size events that can be emitted with a discriminator write plus one body copy

Optional flags can reduce a program further when their tradeoffs are acceptable. `const-rent{:toml}` removes a rent sysvar read from account creation. Disabling `guardrails{:toml}` and `account-resize{:toml}` can reduce size and CU for programs that do not need those safety nets.

The checked-in bench programs live under `bench/programs/`. See [Examples and benchmarks](/docs/v2/reference/examples-and-benchmarks/) for the current table and reproduction commands.
